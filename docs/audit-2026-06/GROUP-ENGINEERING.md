# Staff Engineer Group Digest — MeetingScribe Refactor (Audit 2026-06)

## Executive Summary

The MeetingScribe Refactor codebase (269 Swift files, ~59K LOC) has strong read-path performance, atomic-write discipline, and a principled local-first/privacy posture. But five engineering audits converge on a small set of structural problems that gate the entire roadmap and a handful of live data-loss bugs that are shipping today.

Three themes dominate:

1. **Silent data loss is shipping now.** The live transcript truncates the final 0–5 minutes of every meeting; the ScribeCore daemon recording path is live, auto-registered, and can record an entire meeting into an orphaned, never-finalized folder; and the vault migration is marked complete but only half-wired (no `relativeFolderPath` write-back, forcing O(N) tree walks). These are P0/P1 and largely independent of any refactor.

2. **The two-binary split is scaffolding without substance.** 25 files are physically duplicated between the app and the (never-run-in-production) daemon, and 12 have already diverged — including `MeetingPipelineController` (135 diff lines) and `WhisperRunner` (85). Every recording/transcription fix must land twice, and ENG-A/ENG-D fixes already live in both copies. Extracting a shared **CaptureKit** library and a typed/versioned IPC contract retires this hazard before Phase 2 activation diverges it further.

3. **There is no seam to test the thing that matters most.** 323+ `.shared` singleton accesses mean the record→transcribe→repair→vault-write pipeline has zero end-to-end coverage; the golden-audio and integration test harnesses exist only as fixture-less skeletons. Constructor injection plus completed test fixtures would have caught the live-truncation bug before it shipped.

Cross-cutting reliability fixes (SQLite migration transactions, atomic-write helper, finalize write-ahead journal, DB integrity self-heal) and privacy hardening (plaintext Notion key in Claude config, unvalidated/unsigned IPC commands, iMessage access without audit/consent) round out the slate.

## Top 5 Bets

> **The highest-leverage moves for this group, in order:**
>
> 1. **Fix live-transcript truncation (#1).** This is a shipping P0: every meeting silently loses its final minutes, poisoning summaries and action items. Medium effort, foundational user trust.
>
> 2. **Gate the ScribeCore daemon recording path (#2).** A nondeterministic, total-meeting-loss landmine is live in the shipping binary today. One feature flag stops it immediately and unblocks Phase 2 on our schedule.
>
> 3. **Constructor injection + CaptureKit extraction (#4 + #5).** The two-in-one move that retires the 25-file duplication, kills 323 singleton accesses, and creates the only seam capable of testing the recording pipeline. Everything else in Phase 2 compounds badly without it.
>
> 4. **End-to-end pipeline integration harness + golden-audio suite + real CI gate (#6 + #7 + #8).** Together these would have caught the truncation, orphan-folder, and migration bugs *before* shipping. Turns the skeleton test infrastructure into a real safety net.
>
> 5. **Data-integrity quartet: SQLite migration transactions, insertPerson v3 columns, atomic-write helper, finalize journal (#9 + #10 + #11 + #18).** Small-to-medium fixes that close the silent-corruption and stranded-meeting failure modes that worsen as the vault scales past 200 meetings.
