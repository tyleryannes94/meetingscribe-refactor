# Engineering тАФ Security & Privacy

_Lens: does the "everything stays local" promise actually hold тАФ no silent egress, secrets in the Keychain not plaintext, MCP writes that can't escape the vault, downloads that are verified, and consent that's real?_

## Full-app audit (through my lens)

**The local-first promise mostly holds, and the foundations are good.** Verified by enumerating every outbound host in the source: the only network destinations are opt-in integrations (`accounts.google.com`, `oauth2.googleapis.com`, `people.googleapis.com`, `www.googleapis.com/drive`, `api.notion.com`, `api.linear.app`), the whisper model host (`huggingface.co`), and the Sparkle appcast (`github.com`). **Zero analytics/telemetry SDKs** (grep for sentry/mixpanel/amplitude/posthog/datadog = nothing). The cloud-LLM path is **dead**: `AnthropicClient.swift:126-131` documents that chat now runs entirely against local Ollama; `AnthropicClient(` is constructed in zero non-dead files. Good тАФ but the class, its `api.anthropic.com` endpoint constant (`:18`), and `KeychainStore.Account.anthropicAPIKey` are stale remnants whose presence misleads a security reviewer into thinking transcripts go to Anthropic. Dead crypto/egress surface should be deleted, not left "reserved."

**Secrets are correctly Keychain-backed.** `KeychainStore.swift` is a clean `kSecClassGenericPassword` wrapper with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` (no iCloud sync, not in Time Machine) and a transparent UserDefaultsтЖТKeychain migration that clears the plaintext plist (`:87-95`). `Settings.swift:276-341` routes `notionAPIKey`/`linearAPIKey`/`googleClientSecret`/`googleRefreshToken` through it. This is the single best security decision in the codebase.

**Whisper model integrity is half-done.** ENG-D's SHA-256 pin is real and tested: `WhisperRunner.swift:267` pins `a03779c8тАжd002`, `:382` rejects on mismatch, temp-file + atomic rename prevents a partial install (`:392-401`). **But the consent half of ENG-D was never built.** `:296-303` silently fetches ~140 MB from HuggingFace on the first transcription with no prompt тАФ `tryAutoDownloadBaseEnModel` has zero callers that gate on a consent flag (grep confirms). For an app whose whole pitch is "nothing leaves your Mac," a silent first-run download to a third-party host is exactly the surprise the plan said to avoid. And the pin/fetch logic is **duplicated** in `Sources/ScribeCore/Transcription/WhisperRunner.swift:273/305` тАФ two copies to keep correct (the known ARCH-1 drift hazard, now load-bearing for security).

**MCP write tools are append-only and mostly safe, with one path-trust gap.** The 5 writers (`main.swift:1395-1414`) are sensible: `create_meeting_note` **appends, never overwrites** with a dated "Added by Claude" header (`:1382-1386`); `add_memory` recovers the raw payload without decoding so it can't drop fields (`:1336-1356`); `add_person` slugs the display name through `personSlug` which strips `/:\?%*|"<>` (`:281`). **The gap:** `directoryForMeeting` (`:99-101`) feeds a meeting's stored `relativeFolderPath` straight into `appendingPathComponent(rel)` with no containment check тАФ a corrupt or attacker-influenced `meeting.json` carrying `../../...` would let an MCP write land **outside the vault**. There is no allowlist that the resolved write path is a descendant of `storageDir`. Also: **the MCP server has no auth and trusts `MEETINGSCRIBE_STORAGE` blindly** (`:21`) тАФ any process that can spawn the binary with an env var can point it at an arbitrary tree, and there is **no audit log** of what Claude wrote.

**The iCloud `_inbox` watcher is an unauthenticated ingestion channel.** `iCloudInboxWatcher.swift:94-101` decodes any JSON file that lands in `vault/_inbox/` and auto-routes `type: "add-person"` / action-item / quick-note envelopes into the data store with no signature, sender check, or provenance field. Anything with write access to that iCloud folder (a second signed-in device, a shared-folder collaborator, malware with Drive access) can inject CRM records or tasks. The trust boundary is "whoever can write the folder," which is wider than "the user's iPhone."

**Dictation leaks through the clipboard.** `TextInserter.insertAtCursor` (`:27-55`) implements F5 paste by writing the dictated text to the **system pasteboard** and synthesizing тМШV, restoring the prior clipboard ~0.35 s later. During that window every dictated sentence тАФ potentially passwords or sensitive notes тАФ is readable by any clipboard manager or background app. There's no "concealed"/transient pasteboard flag and no option to use the AX API for direct insertion.

**Ollama URL is user-settable with no egress allowlist.** `Settings.swift:135` defaults to `http://127.0.0.1:11434`, but `SettingsView.swift:376-485` lets the user set any URL, and `OllamaChatClient.swift:130` POSTs the full transcript/summary prompt there. A typo'd or maliciously-set remote URL silently ships every meeting off-device over plain HTTP, and nothing warns that the LLM endpoint left localhost. The "local LLM" promise is only as strong as an unguarded text field.

**Code-signing is self-signed/ad-hoc; not notarized.** `Makefile:121-151` signs with the local "MeetingScribe Local Signer" identity (or ad-hoc `-`), enables hardened runtime, but there is **no `notarytool`/`stapler` step anywhere** (grep confirms). `Entitlements.plist:27` sets `disable-library-validation` for Sparkle. `Info.plist:20` ships `CFBundleVersion = 21ca8db-dirty` тАФ a dirty (uncommitted) build is the installed artifact. The app is **not sandboxed** (`ScribeCore.entitlements:5` `app-sandbox = false`), so the vault-escape and inbox-trust issues above aren't backstopped by the OS.

## Existing-plan items I rank highest

1. **ENG-D (model checksum + consent)** тАФ endorse, but flag it as **only half-shipped**. SHA pin landed; the consent gate the plan explicitly called for did not. Finish the consent half.
2. **ENG-E (backup honesty)** тАФ a privacy-adjacent truth-in-labeling issue: don't claim "backed up"/"encrypted" when it's plain files in iCloud Drive. Users make sensitivity decisions based on that claim.
3. **ARCH-1 (CaptureKit de-dup)** тАФ security-relevant now: the SHA-pin/download logic is duplicated appтЖФdaemon, so a future fix to the download path must be made twice or one copy silently regresses.
4. **AUDIT_REPORT #5 (Sparkle `SUFeedURL` repo)** тАФ endorse: an auto-update feed is a code-execution channel; it must point at the correct, controlled repo (still worth confirming alongside the EdDSA key already present at `Info.plist:46`).

## NET-NEW recommendations

- **E4-1 тАФ Vault containment guard for all MCP/app writes.** `S` ┬╖ **High** ┬╖ no deps. Add a single `resolveInsideVault(_ rel:) -> URL?` that rejects any resolved path not prefixed by `storageDir` (canonicalize, reject `..`). Route `directoryForMeeting` (`main.swift:99`), `peopleRoot`, and every writer through it. Closes the only path-traversal vector in the write-capable MCP. *What/why:* a corrupt or hostile `relativeFolderPath` can currently steer a write outside the vault.

- **E4-2 тАФ MCP write audit log + reversible journal.** `M` ┬╖ **High** ┬╖ depends on E4-1. Every write tool appends a line to `vault/_mcp-audit.jsonl` (tool, args digest, target path, timestamp, before-hash) and snapshots the prior file bytes. Gives the user a "what did Claude change?" view and one-click undo. *User value:* trust that an agent with write access can't quietly rewrite their second brain.

- **E4-3 тАФ Network egress allowlist + "endpoint left localhost" guard.** `S` ┬╖ **High** ┬╖ no deps. Centralize all outbound `URLSession` calls behind an `EgressPolicy` that (a) enforces a compile-time host allowlist and (b) shows a one-time confirm when `ollamaURL` host тИЙ {localhost, 127.0.0.1} before any transcript is POSTed. Turns the implicit "stays local" promise into an enforced invariant. *Why:* `OllamaChatClient.swift:130` will happily ship transcripts to any URL the user typed.

- **E4-4 тАФ First-run model-download consent (finish ENG-D).** `S` ┬╖ **Med** ┬╖ no deps. Gate `tryAutoDownloadBaseEnModel` behind an onboarding "download the 140 MB speech model from huggingface.co?" prompt with a persisted flag; offer a manual/offline path. *Why:* completes the half of ENG-D that shipped only its checksum.

- **E4-5 тАФ Vault encryption at rest (sensitive-meeting mode).** `L` ┬╖ **High** ┬╖ no deps. Per-meeting opt-in: encrypt `transcript.md`/`summary.md`/audio with a key wrapped in the Keychain (or FileVault-style per-vault key), so a synced iCloud copy or a stolen-but-logged-in-elsewhere device can't read it. Pair with a **"sensitive meeting" toggle** that also suppresses cloud-export buttons and MCP read exposure for that meeting. *User value:* the only durable answer to "my whole life's meetings sit in plaintext in iCloud Drive."

- **E4-6 тАФ Secret scanning / redaction before any cloud hop.** `M` ┬╖ **Med** ┬╖ no deps. Before a transcript/summary is handed to Drive export, Notion sync, or a non-local Ollama, run a redaction pass (API keys, emails, card/SSN patterns) and show what will be sent. Even with everything local by default, the export buttons are the leak points. *Why:* there's currently no review step between "local note" and "pushed to Notion."

- **E4-7 тАФ Signed/provenance-checked inbox envelopes.** `M` ┬╖ **Med** ┬╖ no deps. Have the iPhone Shortcuts embed a per-device HMAC (key shared once via QR/pairing) in each `_inbox` envelope; the watcher (`iCloudInboxWatcher.swift:94`) rejects unsigned/unknown-device payloads and quarantines them for review. *Why:* today any writer to the iCloud folder can inject CRM/task records.

- **E4-8 тАФ Consent ledger + privacy dashboard.** `M` ┬╖ **Med** ┬╖ no deps. A single screen that shows, from a `vault/_consent.jsonl` ledger, exactly which TCC permissions are granted (mic, screen, calendar, contacts, automation), which integrations are connected, what was sent off-device and when, and a "revoke/disconnect" for each. Materializes the "everything stays local" claim into something auditable. *User value:* turns trust-by-assertion into trust-by-inspection.

- **E4-9 тАФ Concealed-pasteboard dictation insertion.** `S` ┬╖ **Med** ┬╖ no deps. In `TextInserter`, set `org.nspasteboard.ConcealedType` / `org.nspasteboard.AutoGeneratedType` (transient flags clipboard managers honor) on the dictated string, shorten the restore window, and offer an opt-in direct-AX-insertion path that never touches the pasteboard. *Why:* F5 dictation currently exposes every utterance to clipboard watchers for ~0.35 s.

- **E4-10 тАФ Notarize + staple release builds; block dirty installs.** `S` ┬╖ **Med** ┬╖ no deps. Add a `notarytool`+`stapler` step to the release path and refuse to cut a release whose `CFBundleVersion` ends in `-dirty` (`Info.plist:20`). Self-signed local rebuilds stay fine for dev, but a distributed binary that isn't notarized trains users to bypass Gatekeeper тАФ the opposite of a security posture. *Why:* the Sparkle auto-update channel distributes binaries; unnotarized ones are an attack-surface and a UX foot-gun.

- **E4-11 тАФ Delete the dead Anthropic/cloud-LLM surface.** `S` ┬╖ **Low** ┬╖ no deps. Remove `AnthropicClient`, its endpoint constant, and the `anthropicAPIKey` Keychain account, or move them behind an explicit, off-by-default `#if CLOUD_CHAT`. *Why:* dead egress code with a real API endpoint is a liability тАФ it lies to anyone auditing what the binary can reach, and a future refactor could silently re-activate it.

- **E4-12 тАФ At-rest scrub for revoked integrations.** `S` ┬╖ **Low** ┬╖ depends on E4-8. When the user disconnects Google/Notion/Linear, actually delete the refresh/API token from the Keychain (not just flip an `isConnected` bool) and revoke the OAuth grant server-side where the API allows. *Why:* `GoogleDriveService.init` infers connection from a stored refresh token; "disconnect" should guarantee the token is gone.

## Top 3 picks

1. **E4-3 тАФ Network egress allowlist + "endpoint left localhost" guard.** The single highest-leverage way to make the marquee promise enforceable rather than aspirational, and it directly closes the live `ollamaURL`-can-be-remote hole.
2. **E4-1 тАФ Vault containment guard for all writes.** The one concrete, exploitable path-traversal in the now-write-capable MCP; small, self-contained, high impact.
3. **E4-5 тАФ Vault encryption at rest + sensitive-meeting mode.** The durable answer to "years of meetings sit in plaintext in iCloud Drive" тАФ the biggest residual privacy exposure once egress is locked down.
