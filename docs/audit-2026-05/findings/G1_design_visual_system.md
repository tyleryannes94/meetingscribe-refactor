# Design — Visual Design System

*Lens: typography, spacing, color/theming, dark-mode parity, component consistency, and information density across `Sources/MeetingScribe/UI`.*

## Full-app audit (through my lens)

MeetingScribe has a **design system that exists but is only half-adopted**. `NotionDesign.swift` (`NDS`) is a genuinely good foundation — a warm neutral palette tuned for dark mode, a dynamic light/dark `NSColor` builder (`NotionDesign.swift:55-62`), a real button family (`MSPrimary/Secondary/Danger/Tertiary`, `:227-283`), reusable chips/eyebrows/property-rows, a 44pt `minTap()` extension (`:287-292`), and a Light/Dark `AppearanceToggle` (`:434-468`). This is more mature than most indie macOS apps. The problem is **coverage and drift**, not the tokens themselves.

**1. The token system is adopted in fewer than half the files.** Only **20 of 43** UI files reference `NDS.` at all. The files that *opt out* are the worst ones to lose — the entire ActionItems family (`ActionItemsBoardView`, `ActionItemsListView`, `ActionItemsTableView`), `ChatSidebar`, `SettingsView`, `GlobalSearchView`, `OnboardingSheet`, `UnifiedMeetingDetail`, `MeetingTranscriptTab`, `MenuBarView`, and `NeedsAttentionWidget` use **none** of it. These are the densest, most-used surfaces, so the system's reach is inversely correlated with where it matters.

**2. Two parallel color systems coexist — warm NDS vs. cold AppKit.** `NDS.bg/sidebarBg/fieldBg` are warm near-blacks (#1C1B19 family). But the opted-out files paint surfaces with `Color(NSColor.controlBackgroundColor)` and borders with `Color(NSColor.separatorColor)` — the *cold blue-gray* system surfaces NDS was explicitly created to replace (its own header comment, `:36-38`, says it "replaces the cold blue-navy"). Concretely: `ActionItemsBoardView.swift:56/78/149/151`, `ChatSidebar.swift:33-34/172`, `GlobalSearchView.swift:60`, `QuickNotesView.swift:433`, `ActionItemsProjectPage.swift:377/690`, `TodayView.swift:422/429`. The result is that the Tasks board and chat rail render in a noticeably colder, bluer gray than the rest of the app, and the seams are visible side-by-side.

**3. The brand color is purple, but ~15 surfaces render system blue.** `NDS.brand = #7F56D9` and `MainWindow.swift:262` sets `.tint(NDS.brand)`. But many views hardcode `Color.accentColor` as a *fill/foregroundStyle* — which resolves to the macOS system accent (blue by default), **not** the `.tint`: `NeedsAttentionWidget.swift:50`, `ActionItemsWidget.swift:40/59`, `ChatPanel.swift:67/131`, `ActionItemsTableView.swift:110`, `ActionItemsSidebar.swift:39`, `TaskRowView.swift:114`, `ActionItemsProjectPage.swift:369`, `HotkeyRecorder.swift:29/33`, `OnboardingSheet.swift:191`, `AudioLevelMeter.swift:100`. So the primary action color is purple in Meetings/People and blue in Tasks/Chat/Onboarding. This is the single most visible inconsistency in the app.

**4. No spacing or radius scale — only ad-hoc magic numbers.** `NDS` defines `radius/rowRadius/cardRadius` (8/8/12) but **no spacing token at all**. Searching the UI: **255 hardcoded `.padding(N)` literals**, and corner radii fragment across **10 distinct values** (`8`×27, `6`×17, `10`×8, `14`×7, `7`×6, `12`×6, `9`×4, `4`×3, `26`×3, `0`×1) despite the tokens existing. Cards round at 14 (`cardRadius` is 12), chips at 7, tiles at 9 — none reference the token. There is no `NDS.space` (4/8/12/16/24…) so every view re-invents its rhythm.

**5. Typography is a token set the app routinely ignores.** `NDS` defines 6 type tokens (`title/pageTitle/sectionLabel/body/small/tiny`, `:65-70`). Yet the UI contains **143 raw `.font(.system(size:))` calls** vs only **68 `NDS.<font>` references** — raw sizes outnumber tokens 2:1. Worse, the raw sizes fragment into **~24 distinct point values** (13, 11, 12, 14, 10, 9, 26, 36, 16, 11.5, 32, 30, 12.5, 13.5, 15, 18, 22, 24, 38, 40, 48…). There is no real type *scale* — adjacent screens use 12 vs 12.5 vs 13 for the same role. Even NDS's own components use raw sizes (`NotionChip` uses `11.5`, not `NDS.tiny`).

**6. Dark-mode "parity" is untested where it's needed most.** NDS surfaces resolve correctly via `dyn()`. But the opted-out files lean on `NSColor.controlBackgroundColor`/`separatorColor`, which *do* adapt — so they don't break, they just don't match. The app **defaults to light** (`MainWindow.swift:69`, `appearanceDark = false`) per the comment "we default to false rather than forcing dark," yet NDS's own header says the app "lives" in dark mode and is "tuned for dark." So the polished palette is tuned for the mode the app doesn't ship by default, and the cold-AppKit surfaces are most jarring precisely in the *light* default.

**7. No "System / Auto" appearance option.** `AppearanceToggle` is a hard binary (`Light`/`Dark`, `:434`) driving `.preferredColorScheme(appearanceDark ? .dark : .light)` (`:263`). There is **no way to follow the macOS system appearance** — the app overrides it unconditionally. A Mac user who sunsets to dark at night gets a stuck app. This is a notable platform-convention miss.

**8. Density is fixed and fairly loose.** `NDS.pagePadding = 56` and `notionPageColumn()` adds another 28pt vertical + the `contentMaxWidth = 720` cap. For a CRM/tasks tool that power users live in, there's one density and it's the spacious one — no compact mode. Property rows are fixed at a 132pt label column (`NotionPropertyRow`, `:140`). On the data tables (ActionItems) the spaciousness fights the use case.

**9. Iconography & motion are inconsistent but not broken.** SF Symbols are used throughout (good, native), but weights are ad-hoc (`semibold`×34, `bold`×19, `medium`×15) with no rule for which weight an icon at which size takes. Motion is sparse and mostly fine (`easeOut(0.1)` press states in the button styles), but only ~10 files animate at all and durations vary (`0.1`/`0.12`/`0.18`) with no token — transitions between tabs (the ZStack opacity switch in `MainWindow`) get no shared spring.

## Existing-plan items I rank highest (through my lens)

1. **LAY-4 — migrate cold `NSColor.*` surfaces to NDS tokens; adopt `MSPrimary/Secondary/Tertiary` consistently.** The plan files it as **P2**; through a visual-system lens it is the *most* important visual item in the whole plan. It is the direct fix for findings #1–#3 (two color systems, blue-vs-purple). It should be **P1**. Endorse strongly, reprioritize.
2. **LAY-1 — relax the 720/920 width caps, keep ~720 only for prose.** Correct instinct; the cap is currently applied to *tables and lists* (`notionPageColumn`) where it wastes the most space. Endorse.
3. **LAY-3 — replace the People `.padding(.top, 60)` magic number with layout-aware insets.** Already partly done via `NDS.splitPaneTopInset`, but it's the canary for finding #4 (no spacing scale) — endorse extending the principle app-wide.
4. **LAY-2 — chat rail closed by default.** Endorse: as a layout/density default it directly affects perceived information density on first run.

## NET-NEW recommendations

**D2-1 — Add a spacing scale to NDS (`space.xs…xxl`) and a corner-radius rule, then lint for literals.** *(What/why)* There is no spacing token; 255 padding literals and 10 distinct radii. Define `NDS.space` (4/8/12/16/24/32) and standardize radii to the existing 8/12 tokens, then add a tiny CI grep that fails on new `cornerRadius: <literal>` and `.padding(<literal>)` in `UI/`. *(Value)* Visual rhythm becomes consistent; new code can't regress. *(Effort M · Impact High · depends on nothing.)*

**D2-2 — Collapse the type sprawl into a real scale and ban raw `.system(size:)` in UI.** *(What/why)* 143 raw font calls across ~24 sizes vs 68 token uses. Add the missing roles NDS lacks (`caption2`, `display`, `mono`) so devs stop reaching for raw sizes, then migrate the top offenders (ActionItems, Settings, Chat) and add a lint. *(Value)* One coherent hierarchy; cross-screen consistency. *(Effort M · Impact High · depends on nothing.)*

**D2-3 — Unify the accent: replace every `Color.accentColor` *fill* with `NDS.brand`, OR set the app's `NSApplication` accent to brand.** *(What/why)* Finding #3 — ~15 surfaces render system blue against a purple-branded app. Either swap the literals (precise) or set the accent at the app level so `accentColor` resolves to purple everywhere. *(Value)* The brand finally reads as one color. *(Effort S · Impact High · depends on nothing.)*

**D2-4 — Add a "System / Auto" appearance mode.** *(What/why)* Finding #7 — the binary toggle force-overrides macOS appearance with no follow-system option. Make `AppearanceToggle` a 3-segment Auto/Light/Dark and drop `.preferredColorScheme` to `nil` for Auto. *(Value)* Respects the platform convention; fixes the "stuck in light at night" case. *(Effort S · Impact Med · depends on nothing.)*

**D2-5 — Ship a `compact` density mode for data-dense surfaces.** *(What/why)* Finding #8 — one loose density. Add `NDS.density` (`comfortable`/`compact`) that scales row heights, padding, and the `NotionPropertyRow` label column; default ActionItems tables and the People list to compact. *(Value)* Power users see more per screen; the spacious default stays for reading surfaces. *(Effort M · Impact Med · depends on D2-1.)*

**D2-6 — Establish a card/list/surface component contract so the opted-out files can't drift.** *(What/why)* Finding #1 — ActionItems/Chat/Settings re-implement cards from raw `Color(NSColor.controlBackgroundColor)`. Extract `MSCard`, `MSListRow`, `MSSurface`, `MSPanel` views that bake in NDS bg/hairline/radius, and replace the hand-rolled backgrounds. *(Value)* New surfaces inherit the system for free; eliminates the cold-vs-warm seam. *(Effort M · Impact High · depends on D2-1.)*

**D2-7 — Define an elevation/shadow token set.** *(What/why)* Shadows are ad-hoc (`MSPrimary` has `radius:6,y:2`; `AppearanceToggle` has a one-off light-mode shadow; most surfaces have none). Add `NDS.elevation(0…2)` for flat/raised/floating and apply to cards, popovers, and the floating overlay. *(Value)* Consistent depth language; the floating overlay and menu-bar popover stop looking flat against content. *(Effort S · Impact Med · depends on nothing.)*

**D2-8 — Codify an icon weight/size rule and an icon-token map.** *(What/why)* Finding #9 — icon weights are random. Define `NDS.icon(size:role:)` returning the correct SF Symbol weight per size (e.g. 13pt→medium, 16pt→semibold), and a small map of semantic icons (record, person, task, calendar) so the same concept never uses two glyphs. *(Value)* Visual consistency + a single place to swap glyphs. *(Effort S · Impact Med · depends on nothing.)*

**D2-9 — Add a `MSStatusBadge`/semantic-color layer over `selectColor`.** *(What/why)* `NDS.selectColor` hash-maps arbitrary strings to colors with a few semantic overrides (`:87-100`), so "overdue"/"done"/"urgent" colors are buried in a switch and re-derived per call site (`TaskRowView` rolls its own due-date colors at `:297-300`). Promote semantic status colors (`success/warning/danger/info/neutral`) to first-class tokens and route badges through one component. *(Value)* Status color means the same thing everywhere; accessible contrast can be tuned once. *(Effort S · Impact Med · depends on nothing.)*

**D2-10 — Motion tokens + a shared tab transition.** *(What/why)* Durations vary (`0.1/0.12/0.18`) with no token, and the `MainWindow` tab switch is a bare opacity ZStack with no shared motion. Add `NDS.motion` (`fast/standard/slow` + a standard spring) and a single navigation transition. *(Value)* Cohesive feel; respects `reduceMotion` in one place. *(Effort S · Impact Low–Med · depends on nothing.)*

**D2-11 — Snapshot tests for light & dark parity on the core surfaces.** *(What/why)* Findings #2/#6 — there's no guard that a screen looks right in *both* modes; the cold-AppKit surfaces only differ subtly so they slip through. Add SwiftUI snapshot tests rendering Today/Meeting-detail/Tasks-board/People in both schemes and diff against goldens in CI (pairs with the plan's TST-1). *(Value)* Light/dark regressions get caught mechanically. *(Effort M · Impact Med · depends on TST-1 CI.)*

**D2-12 — A living "Design System" debug screen.** *(What/why)* The team has no canonical render of all tokens/components, which is *why* drift happened. Add a hidden gallery view (every color, type token, button, chip, card, badge in both schemes and both densities). *(Value)* Makes drift visible, speeds review, onboards future contributors. *(Effort S · Impact Med · depends on D2-1/D2-6 to be worth most.)*

## Top 3 picks

1. **D2-6 — Component contract (`MSCard/MSListRow/MSSurface`)** — kills the root cause of the cold-vs-warm, two-color-system drift by giving the opted-out files a system-backed surface to use; everything else compounds on it.
2. **D2-3 — Unify the accent (purple, not system blue)** — smallest effort, highest visible payoff; today the brand color literally changes between tabs.
3. **D2-1 — Spacing + radius scale with a lint** — converts the design system from "tokens that exist" to "tokens that are enforced," stopping the 255-literal/10-radius sprawl from growing.
