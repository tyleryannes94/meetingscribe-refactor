# Engineering — Reliability & Data Integrity

> Lens: the meeting record is irreplaceable. Hunt for any path that loses or corrupts data, fails silently, or leaves inconsistent state after a crash or interrupted operation. Endorse the integrity items that matter, then find NEW failure modes the plans miss.

## Full-app audit (through my lens)

I read the live source at `~/MeetingScribeRefactor` and confirmed the headline fixes the plans claim are genuinely in the tree:

- **ENG-A is solid.** `LiveTranscriber.flush()` (`LiveTranscriber.swift:155`) awaits the per-source task chains before render; `MeetingManager.stopRecording` calls it (`MeetingManager.swift:339`) and snapshots `droppedChunkCount` / `lastTranscribedSecond` / `recordedDuration` (`:363-365`); `MeetingPipelineController.needsBatchRepair` gates the batch repair on empty OR `droppedChunks>0` OR short coverage (`MeetingPipelineController.swift:65-74`). The dropped-chunk safety net is real: backpressure drops only the *live-preview WAV* (`LiveTranscriber.swift:71`), never the persistent `mic-NNN.m4a` segment, so the batch pass recovers it. Good.
- **ENG-B is mostly addressed but has a residual gap (see E3-2).** `desiredDirectory` now emits the date-partitioned layout (`MeetingStore.swift:101`), the tree-walk goes 3 deep (`:587-602`), and the migration only sets the completed flag when `failures==0` (`VaultMigrationManager.swift:107`).
- **ENG-C is half-done.** The synchronous `beginPipeline` claim closes the Transcribe-Now race (`MeetingManager.swift:353`, `MeetingPipelineController.swift:227`) — but the *second half of ENG-C, "make `mergeSegments` write temp + atomic rename," was never done* (see E3-3).
- Crash recovery is well-built: `.recording.inprogress` markers, launch sweep, manifest rebuild, iCloud-eviction-tolerant discovery (`AudioRecovery.swift`).
- Writes go through `NSFileCoordinator` + `.atomic` (`MeetingStore.coordinatedWrite`, `:179-200`); SQLite is correctly in Application Support (`SecondBrainDB.dbURL:38`).

But the reliability picture is **not** as clean as `AUDIT_REPORT_2026-05-30.md` implies. The biggest issue is one the existing docs explicitly dismiss as "Phase 2, deferred, not a regression."

### CRITICAL — the ScribeCore daemon path is a live data-loss landmine, shipping today

The plans treat the two-binary split as inert scaffolding. It is not inert — it is **wired and active**:

1. `MeetingScribeApp.startServices()` calls `registerScribeCoreLoginItem()` **unconditionally** at every launch (`MeetingScribeApp.swift:152`), and `Makefile:84` (`app: scribecore`) + `:114-118` embed `ScribeCore.app` in `Contents/Library/LoginItems/`. So the daemon **is shipped and auto-registered to boot on login** with no feature flag.
2. On boot the daemon posts `coreReady` (`ScribeCoreServices.swift:44`). The UI's `tryStartViaScribeCore()` then succeeds and sets `usingScribeCore = true` (`MeetingManager.swift:192-193`), routing the *next recording* to the daemon.
3. The daemon's `handleStartRecording()` captures audio into a **brand-new orphan directory** `meetings/scribecore-<date>/` (`ScribeCoreServices.swift:65-66`) that has **no `meeting.json`**, is divorced from the UI's `activeMeeting` folder, and **never wires `onMicChunk`/`onSystemChunk` to any LiveTranscriber** (compare the direct path, `MeetingManager.swift:197-202`).
4. On stop, the daemon just posts `recordingStopped` (`ScribeCoreServices.swift:88`). The UI observer (`MeetingManager.swift:135-149`) writes `liveTranscriber.renderMarkdown()` — which is **empty**, because nothing fed the live transcriber — into the UI's `activeMeeting` folder (which received **zero audio**), then sets state idle. **It never calls `pipelineController.finalize()`**: no merge, no batch transcribe, no summary, no action items, no FTS index, no markdown.

Net effect when the daemon happens to be up first: the user records a meeting, sees the timer run, hits stop — and ends up with an empty transcript on a meeting with no audio, while the actual audio sits in an unreferenced `scribecore-*` folder the app never surfaces. **This is silent, total loss of a meeting.** It's nondeterministic (depends on whether `coreReady` arrived before the user hit record), which makes it worse — it'll look like a flaky "sometimes my recording is blank" bug.

The mitigating fact is that in a fresh single-binary mental model the daemon may not be permitted/launched, and the 5s `coreReady` wait often times out on the first recording. But "often" is not "never," and the login item is registered, so this can fire. The existing audit's claim that the `usingScribeCore` path "already exists and does it right" (V3 ENG-F note; AUDIT_REPORT §2) is **incorrect for the daemon side**.

### Secondary integrity findings

- **`mergeSegments` is non-atomic** (`AudioRecorder.swift:367`): it `removeItem(at: output)` the existing `mic.m4a`/`system.m4a`, then writes the new merge in place. A crash or `PassthroughAudioMerger` failure mid-merge leaves the merged file absent/truncated. Recoverable via Transcribe-Now (segments survive), but the in-flight merged artifact is corruptible — ENG-C explicitly asked for temp+rename and it wasn't done.
- **ENG-B residual: migration never rewrites `relativeFolderPath`.** `VaultMigrationManager.migrateLayout` moves folders (`:89`) but never updates each `meeting.json`'s stored `relativeFolderPath`, and never refreshes `.meeting-index.json` or `_recent.json`. After migration, every meeting's persisted path is stale → `directory(for:)` fast-path misses (`MeetingStore.swift:118-124`) and falls to the O(N) tree-walk on first access for *every* migrated meeting, and `_recent.json` (read by iPhone Shortcuts) points at dead paths until each meeting is individually re-saved. The tree-walk self-heals one-at-a-time, so no data is lost, but it's an O(N²)-ish first-run stall and a stale-index window.
- **No SQLite integrity verification anywhere.** `grep` for `integrity_check`/`quick_check`/`SQLITE_CORRUPT`/`wal_checkpoint` returns nothing. `SecondBrainDB.open()` (`:48`) opens WAL and runs `ensureSchema` but never validates the file. A truncated/corrupt `secondbrain.db` (power loss during WAL checkpoint, disk-full) will silently return empty search results — the People graph (the product's moat) goes dark with no signal and no auto-rebuild. `rebuild()` wraps work in `BEGIN;`/`COMMIT;` (`:166-174`) with **no rollback on a failed step**, so a mid-rebuild failure can leave the index half-populated and *committed*.
- **`try?` still swallows several user-facing persistence writes (ENG-G partial).** `MeetingManager.swift:583` (`setTranscript` — a *manually imported* transcript silently dropped on write failure), `MeetingPipelineController.swift:300` (transcribeNow summary), and the entire QuickNotes write surface (`QuickNotesController.swift:86,113,165-166,195,221,234` — voice notes and their transcripts vanish silently on write failure). The finalize pipeline got real error handling; these adjacent paths did not.
- **`writeIndex` is fire-and-forget on a detached task** (`MeetingStore.swift:471-489`): the in-memory cache is updated synchronously but the disk write is best-effort `try?`. If the app is killed in the ~ms window after a meeting write but before the index task lands, `.meeting-index.json` is stale; recovered only by a full rescan, which only happens when the index file is *missing*, not when it's stale. A meeting can be invisible in the list until a forced rebuild.
- **iCloud conflict resolution is absent.** `coordinatedWrite` handles single-writer atomicity, but there is no handling of iCloud `.icloud`-conflict sibling files (e.g. `meeting 2.json`, `transcript 2.md`) that iCloud Drive creates when two Macs edit the same vault. Nothing detects or merges them; the app will simply read whichever `meeting.json` it resolves and ignore the conflict copy — a silent divergence/loss across devices, which is exactly the scenario the iCloud-vault strategy invites.

## Existing-plan items I rank highest (through my lens)

1. **ENG-C atomic `mergeSegments`** — the one piece of ENG-C that was skipped. Cheap, removes a real corruption window on the canonical merged audio. (V3 ENG-C, S)
2. **ENG-G — replace remaining `try?` on persistence** — finish the job; voice notes and imported transcripts currently fail silently. The finalize path proves the pattern (`ErrorReporter` + `lastError`); apply it to QuickNotes + `setTranscript`. (V3 ENG-G, M)
3. **ARCH-1 CaptureKit extraction** — not cosmetic through this lens: ENG-A and the merge logic live in *two diverging copies* (app + daemon). The daemon's copy is the one that's broken (above). Until there is one implementation, every integrity fix must be made twice and the daemon will keep drifting. (V3 ARCH-1 / REMAINING_WORK §1, L)
4. **TST-2 data-integrity tests** — specifically a record→stop→finalize end-to-end test that asserts a non-empty transcript and an audio file in the *referenced* folder. Such a test would have caught the daemon orphan-folder bug. (V3 TST-2, M)

## NET-NEW recommendations

**E3-1 — Gate (or disable) the ScribeCore login-item + daemon recording path until it actually finalizes.** *What/why:* Stop the silent-data-loss landmine above. Minimum: don't `registerScribeCoreLoginItem()` unless a `usingScribeCore` feature flag (default off) is set, and make `tryStartViaScribeCore()` hard-return false while the flag is off. Proper fix: have the daemon record into the **UI-chosen meeting directory** (pass the dir + segment via the start command), wire its `AudioRecorder` callbacks to a transcriber, and drive the UI's `recordingStopped` handler through the *same* `pipelineController.finalize()` entry point with ENG-A values threaded across the boundary (REMAINING_WORK §2 flags this thread-through but understates that finalize isn't called at all today). *User value:* eliminates "my recording came out blank." *Effort:* S to gate, L to fix properly. *Impact:* **High.** *Depends on:* ARCH-1 to avoid fixing it twice.

**E3-2 — Migration write-back + index reconciliation.** *What/why:* In `VaultMigrationManager.migrateLayout`, after each `moveItem` rewrite that meeting's `meeting.json` with the new `relativeFolderPath`, and at the end call `store.rebuildIndex()` + `updateRecentJSON()` so `.meeting-index.json` and `_recent.json` match disk. *User value:* no post-migration O(N) stalls; iPhone Shortcuts see valid paths immediately. *Effort:* S. *Impact:* Med. *Depends on:* none.

**E3-3 — Pipeline write-ahead journal / finalize state machine.** *What/why:* Today finalize is a linear sequence of independent writes (transcript → summary → meeting.json → markdown → FTS). A crash between steps leaves a meeting in a partial state with no record of *where* it stopped, and nothing re-runs it. Add a tiny `pipeline-state.json` per meeting (`{stage: merged|transcribed|summarized|indexed, updatedAt}`) written before each stage; on launch, sweep for meetings whose stage < `indexed` and resume finalize from there (analogous to the existing `.recording.inprogress` crash sweep, which only covers *capture*, not *finalize*). *User value:* a crash during the (multi-minute, CPU-heavy) summarize step no longer strands a meeting with a transcript but no summary/action-items forever. *Effort:* M. *Impact:* High. *Depends on:* none.

**E3-4 — SQLite integrity check + auto-rebuild on open.** *What/why:* On `SecondBrainDB.open()`, run `PRAGMA quick_check` (and detect `SQLITE_CORRUPT` from any query); on failure, delete the DB and trigger a full `rebuild()` from canonical JSON (which is safe — the DB is derived). Also wrap `rebuild()` so a failed step does `ROLLBACK` instead of leaving a committed half-index. *User value:* the People graph / global search self-heals instead of silently going blank after a bad shutdown. *Effort:* S. *Impact:* High. *Depends on:* none.

**E3-5 — iCloud conflict detector + resolver.** *What/why:* Add a launch + `NSMetadataQuery` sweep for iCloud conflict siblings (`NSFileVersion.unresolvedConflictVersions(of:)`) on `meeting.json`/`transcript.md`/`summary.md`. Surface a "this meeting was edited on two devices" banner and auto-resolve by keeping the longest transcript / newest meeting.json, archiving the loser into the meeting folder rather than deleting it. *User value:* multi-Mac users (the explicit iCloud-vault use case) don't silently lose edits or end up with phantom `meeting 2.json`. *Effort:* M. *Impact:* High. *Depends on:* none.

**E3-6 — "Recover from segments" tooling + integrity self-scan.** *What/why:* Generalize the existing crash sweep into a user-runnable **Vault Doctor**: scan every meeting folder for (a) audio segments but no transcript, (b) `meeting.json` decode failures, (c) merged `mic.m4a` smaller than the sum of segments (truncated merge from E3-3 / non-atomic merge), (d) meetings on disk missing from the index. Offer one-click re-transcribe / re-index / repair. *User value:* a single "is my data OK?" button; turns silent corruption into a visible, fixable list. *Effort:* M. *Impact:* High. *Depends on:* E3-4 helpers.

**E3-7 — Atomic write helper for ALL canonical artifacts incl. merged audio + index.** *What/why:* `coordinatedWrite` is atomic for text/JSON, but `mergeSegments` (E3-3 endorse) and the audio import/convert paths (`MeetingManager.convertToM4A:834` does `removeItem` then write) are not. Add a single `writeAtomicallyReplacing(tempIn:finalAt:)` (write to `.<name>.tmp` in same dir, `FileManager.replaceItemAt`) and route merged audio + afconvert output + `_recent.json` + `.meeting-index.json` through it. *User value:* no half-written canonical file can ever be observed. *Effort:* S. *Impact:* Med. *Depends on:* none.

**E3-8 — Local versioned snapshot / "time machine for meetings" before destructive ops.** *What/why:* Before any in-place rewrite of a transcript/summary (Transcribe-Now, re-summarize, import-transcript-over-existing) or a tag-move, copy the prior `transcript.md`/`summary.md` into a `.history/` subfolder with a timestamp (cap N versions). The plans assume iCloud Drive is the backup, but iCloud Drive replicates *deletions and overwrites* — it is not version history for an accidental "Transcribe Now" that replaces a good manual transcript with a worse machine one. *User value:* undo for the operations most likely to destroy a good transcript. *Effort:* M. *Impact:* Med. *Depends on:* E3-7.

**E3-9 — Backup honesty + one-click export bundle (finish ENG-E with teeth).** *What/why:* V3 ENG-E says "don't claim backed-up." Go further: add a "Export vault backup" action that zips the vault (or rsyncs to a user-chosen folder) and records `lastBackupAt`, plus a gentle nag if no backup + not in iCloud Drive. *User value:* a real, verifiable recovery story instead of an implicit one. *Effort:* M. *Impact:* Med. *Depends on:* none.

**E3-10 — Write-failure surfacing for QuickNotes + import (close ENG-G fully).** *What/why:* Replace the `try?` voice-note/transcript writes (`QuickNotesController.swift:86,113,165-166,195,221,234`, `MeetingManager.swift:583`) with reported errors and a per-note error badge (the controller already has an `errors` dict). *User value:* a voice note that fails to save tells you, instead of vanishing. *Effort:* S. *Impact:* Med. *Depends on:* none.

## Top 3 picks

1. **E3-1 — Gate/fix the ScribeCore daemon recording path.** This is the only finding that causes *total, silent loss of an entire meeting* in the shipping binary, and it's nondeterministic. Gate it behind an off-by-default flag *today* (S), schedule the real fix with ARCH-1.
2. **E3-3 — Pipeline write-ahead journal + finalize resume.** The crash sweep covers capture but not the multi-minute finalize. A journal turns "crashed during summarize → meeting stranded forever" into "resumes on next launch."
3. **E3-4 — SQLite integrity check + auto-rebuild.** Cheap, high-leverage protection for the relationship graph that the whole product is built around; today a single bad shutdown can silently empty it with no recovery.
