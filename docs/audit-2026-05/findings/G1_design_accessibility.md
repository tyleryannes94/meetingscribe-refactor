# Design — Accessibility & Inclusive Design

> Lens: VoiceOver, keyboard-only operation, color contrast, Dynamic Type / text scaling, reduce-motion, and internationalization readiness. A meeting-transcription app is *itself* assistive tech (captions for deaf/HoH users) — so its own a11y bar should be high.

## Full-app audit (through my lens)

Coverage today is **sparse and inconsistent**. Hard numbers from the live source:

- **VoiceOver:** only **36** accessibility-modifier usages (`accessibilityLabel/Hint/Value/Element/AddTraits/Hidden`) across **16** files — for a ~199-file, ~43K-LOC SwiftUI app. The one well-handled component is `NotionIconButton`, which maps `.help` → `.accessibilityLabel` and falls back to a humanized symbol name (`NotionDesign.swift:184-189`). That pattern is the exception, not the rule.
- **Localization / i18n: zero.** `NSLocalizedString` / `String(localized:)` / `LocalizedStringKey` / `Localizable` = **0 matches** repo-wide. Every user-facing string is a hard-coded Swift literal (e.g. `"No transcript"`, `MeetingTranscriptTab.swift:33`; `"Transcribing"/"Ready"`, `MeetingCard.swift:248/257`). There is no `.lproj`, no string catalog, and English-only pluralization is hand-rolled (`"attendee\(count == 1 ? "" : "s")"`, `MeetingCard.swift:133`) — which breaks in every non-English locale and isn't even VoiceOver-friendly.
- **Reduce-motion: not handled.** A targeted grep for `reduceMotion`/`reduceTransparency` returns **0 matches**. Yet the app animates aggressively and *perpetually*: `StatusPulseDot` runs `.repeatForever(autoreverses:)` opacity pulse (`MeetingCard.swift:337`); the live card uses `symbolEffect(.pulse, options: .repeating)` (`MeetingCard.swift:70`); cards spring-scale on hover (`:54`); chevrons spring-rotate (`:283`); `QuickActionCard`/`QuickPill` animate on hover. None are gated on `@Environment(\.accessibilityReduceMotion)`. For users with vestibular disorders this is a continuous, un-pausable motion source on the primary surface.
- **Dynamic Type / text scaling: largely defeated.** **192** `.font(.system(size:))` calls across 34 files. The design system itself bakes in fixed point sizes — `NDS.title = 32`, `body = 14`, `tiny = 11` (`NotionDesign.swift:65-70`) — instead of semantic styles (`.title`, `.body`, `.caption`) that respond to the system text-size setting and Larger Accessibility Sizes. macOS honors `.font(.body)`-style semantic fonts under "Larger Text"; fixed pixel sizes do not scale. The 11–11.5pt chip/eyebrow text (`NotionChip`, `:118`) is below comfortable reading size and cannot be enlarged.
- **Color contrast risks.** The warm-neutral dark palette leans on very low-alpha text and fills:
  - `textTertiary = (210,204,190, 0.42)` on `bg (28,27,25)` (`NotionDesign.swift:48`) — effective luminance is well under WCAG AA 4.5:1 for the small (11–12pt) text it's used on (eyebrows, durations, meta).
  - `textSecondary` at `0.72` alpha is borderline at small sizes.
  - Status/health is partly **color-only**: green "Ready" vs orange "Transcribing" chips (`MeetingCard.swift:248-262`) — these *do* pair color with a text label (good), but the bare 6pt colored `Circle()` dots and the `record.circle.fill` red dot carry meaning with no text/shape differentiation for color-blind users. `differentiateWithoutColor` is never consulted.
  - No `increaseContrast` handling anywhere; hairline borders (`hairline` alpha 0.13) effectively vanish under High Contrast.
- **Keyboard-only operation: partial.** Global tab/record shortcuts exist (`MeetingScribeApp.swift:72-111`, ⌘1–5, ⌘R, ⌘N). But the **Meetings list is mouse-only**: `meetingList` is a `ForEach` of plain `Button`s inside a `ScrollView` (`MeetingsView.swift:144-168`), and the month grid likewise (`:374`) — there is no `List(selection:)`, so ↑/↓/Enter don't work (this is the open DEF-2 item). People and Notes *do* use `List(selection:)` (`PeopleListView.swift:166`, `QuickNotesView.swift:55`), so focus behavior is inconsistent across tabs. The `MeetingCard` is a single composite `Button` with no `accessibilityLabel`, so VoiceOver reads a stream of fragments ("9:30 AM, 30 m, Standup, 4 attendees, video, chevron right") instead of one coherent element.
- **Transcription as an accessibility asset — under-exploited.** The transcript is the app's biggest inclusive-design win (live captions + searchable record for deaf/HoH users), but it's treated as a passive document: `TranscriptSyncView` for past meetings, a `LiveTranscriptScroll` while recording (`MeetingTranscriptTab.swift:21-39`). There's no caption-optimized presentation, no font/contrast controls on the transcript reader, and the live transcript isn't surfaced as an always-visible caption band.

## Existing-plan items I rank highest

1. **DEF-2 — make the Meetings list a focusable `List(selection:)`** (`MASTER_PLAN_V3.md` §3.5). Through my lens this is *the* keyboard-access fix: it gives arrow-key/Enter navigation and makes each row a proper VoiceOver list item with traits. Endorse strongly, and extend it to the month grid.
2. **LAY-1 — relax the 720/920 width caps** (§3.2). Critical for low-vision users who run large system text: fixed narrow columns + enlarged text = severe truncation/clipping. Full-width reflow is an a11y enabler, not just aesthetics.
3. **Speaker-labeled transcript & diarization surfacing** (§4 / `REMAINING_WORK.md` §4). For deaf/HoH users a transcript without speaker attribution is far less usable; "who said what" is core caption quality. Endorse and reframe as an accessibility feature, not just a nicety.
4. **NAV-1/NAV-2 — kill inline expand/collapse → click-into + back** (§3.1). A single canonical, navigable detail surface is dramatically easier to reason about for screen-reader and keyboard users than three coexisting open-meeting models.
5. **First-run "how it works" overview** (§5). Onboarding is where assistive-tech users decide whether the app is usable; a VoiceOver-narrated overview pays off here.

## NET-NEW recommendations

**D5-1 — Reduce-motion compliance pass (design-system level).** What/why: read `@Environment(\.accessibilityReduceMotion)` once in the NDS layer; expose an `NDS.motion(_ animation:)` helper that returns `nil` when reduce-motion is on, and gate `StatusPulseDot`, the live `symbolEffect(.pulse)`, hover spring-scale, and chevron rotation on it. Replace perpetual pulses with a static colored state when reduce-motion is set. User value: removes a continuous vestibular trigger on the main screen; table-stakes for App Store accessibility. Effort: S. Impact: High. Depends on: none.

**D5-2 — Dynamic Type adoption via semantic + scalable fonts.** What/why: convert `NDS` type tokens from `.system(size:)` to relative/semantic fonts (`.system(.body)`, `.relativeTo:` or `@ScaledMetric`), and migrate the 192 inline `.font(.system(size:))` call-sites to NDS tokens so text honors the system text-size and Larger Accessibility Sizes. Enforce a minimum body size and let chips/eyebrows scale. User value: low-vision users can actually read the app without a magnifier. Effort: L (mechanical but broad). Impact: High. Depends on: LAY-1 (width caps must relax or enlarged text clips).

**D5-3 — Contrast audit + High-Contrast variants of NDS tokens.** What/why: re-tune `textTertiary` (0.42→~0.62) and `textSecondary`, verify every text/background pair hits WCAG AA 4.5:1 (3:1 for ≥18pt) in both appearances, and add `accessibilityDisplayShouldIncreaseContrast`-aware overrides that thicken hairlines and raise text alpha. Ship a tiny contrast-checker unit test over the NDS palette so regressions fail CI. User value: legible for low-vision and bright-environment users; testable guarantee. Effort: M. Impact: High. Depends on: none.

**D5-4 — Color-independent status (differentiate-without-color).** What/why: the green/orange/red status dots and chips encode state in hue. Add a glyph or shape to each (✓ Ready, ◐ Transcribing, ⚠ failed), and consult `@Environment(\.accessibilityDifferentiateWithoutColor)` to force glyphs on. Apply to `MeetingCard.pastStatus`, `MeetingHealthBadge`, audio-level meters, and tag chips. User value: color-blind users (≈8% of men) can read status. Effort: S. Impact: Med. Depends on: none.

**D5-5 — VoiceOver rotor-ready meeting/person rows.** What/why: wrap each `MeetingCard` and person row in `accessibilityElement(children: .ignore)` with a composed `accessibilityLabel` ("Standup, 9:30 AM, 30 minutes, 4 attendees, transcript ready"), an `accessibilityValue` for status, `.accessibilityAddTraits(.isButton)`, and an `accessibilityHint("Opens meeting")`. Add custom `accessibilityActions` for the inline buttons (Join & Record, Stop) so VO users don't have to hunt nested controls. User value: turns a fragment-soup into navigable, actionable list items. Effort: M. Impact: High. Depends on: D5-? none, but pairs with DEF-2.

**D5-6 — Live Caption Mode (transcription-as-assistive-tech).** What/why: a dedicated, detachable caption band that renders the live transcript in large, high-contrast, user-sizable text with speaker labels — pinnable as a floating always-on-top window during a meeting (reuse `FloatingOverlay`). Optional auto-scroll-lock and a "caption font size" control. User value: makes MeetingScribe a genuine real-time captioning tool for deaf/HoH users in in-person meetings — a category-defining, marketable accessibility feature no competitor frames this way. Effort: M–L. Impact: High. Depends on: live-transcript reliability (ENG-A, done).

**D5-7 — Internationalization scaffolding (string catalog + locale-correct formatting).** What/why: introduce a `String Catalog` (.xcstrings), wrap user-facing literals in `String(localized:)`, replace hand-rolled plurals with stringsdict/automatic pluralization, and replace the hard-coded `DateFormatter(dateFormat: "h:mm a")` (`MeetingCard.swift:309`) with locale-aware `Date.FormatStyle`. User value: even English-only users benefit from correct plurals and locale time/date; unblocks future localization and is a VoiceOver-pronunciation win. Effort: L. Impact: Med. Depends on: none (do early — retrofitting later is far costlier).

**D5-8 — Accessibility settings panel.** What/why: a Settings → Accessibility section exposing app-level toggles: caption font size, "reduce app motion" (independent of system), "always show status glyphs," and "high-contrast UI." Mirrors macOS conventions and gives users control even when they don't want to change global system settings. User value: discoverable, granular control. Effort: M. Impact: Med. Depends on: D5-1/D5-3/D5-4 (consumes their plumbing).

**D5-9 — Keyboard map + focus-order pass beyond Meetings.** What/why: DEF-2 fixes Meetings; extend the same `List(selection:)`/focus treatment to the ActionItems board (drag-only today) and the month grid, add a discoverable "Keyboard Shortcuts" help sheet (⌘/), and ensure the chat rail and floating overlay are reachable and dismissible by keyboard. User value: full keyboard-only operability across every surface. Effort: M. Impact: Med. Depends on: DEF-2.

**D5-10 — Transcript reader accessibility controls.** What/why: add per-reader font-size and contrast controls to `TranscriptSyncView`, ensure timestamp/speaker structure is exposed as VoiceOver headings (`.accessibilityAddTraits(.isHeader)`), and make tapping a line announce speaker + time. User value: the most-read long-form surface becomes navigable by heading and resizable. Effort: M. Impact: Med. Depends on: speaker-diarization surfacing.

**D5-11 — VoiceOver onboarding + permission-flow audit.** What/why: the Screen-Recording "quit and relaunch" step (flagged in §5) is confusing even sighted; for VO users an un-narrated System-Settings hand-off is a dead end. Add accessibilityLabels/hints to `OnboardingSheet`, ensure focus moves to each new step, and narrate the relaunch affordance. User value: assistive-tech users can complete setup unaided. Effort: S–M. Impact: Med. Depends on: none.

**D5-12 — Audio-level meter as accessible value.** What/why: `AudioLevelMeter` (mic/system bars, `MeetingCard.swift:170-178`) is purely visual — a deaf user monitoring "is it capturing?" gets nothing. Expose `accessibilityValue` ("microphone level: high") and an `accessibilityLabel`, and consider a haptic/textual "capturing" confirmation. User value: confirms recording health non-visually and non-aurally. Effort: S. Impact: Low–Med. Depends on: none.

## Top 3 picks

1. **D5-6 — Live Caption Mode.** Highest-conviction net-new: it converts the app's existing transcription engine into a flagship real-time captioning tool for deaf/HoH users. Differentiated, marketable, and leverages work already done (ENG-A).
2. **D5-1 — Reduce-motion compliance pass.** Smallest effort, removes a continuous vestibular trigger that's currently always-on on the primary screen; pure win.
3. **D5-2 — Dynamic Type adoption.** The single biggest everyday-accessibility gap — 192 fixed font sizes lock out low-vision users entirely. Large but mechanical, and it's the difference between "usable" and "unusable" for that audience.

**Single highest-priority recommendation overall:** D5-2 (Dynamic Type / scalable fonts). Fixed point sizes silently exclude every user who relies on enlarged text — the largest accessibility population — and the longer the 192 call-sites accrue, the costlier the fix. Do it alongside LAY-1 so enlarged text has room to reflow.
