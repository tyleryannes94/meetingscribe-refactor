# Sync & State Propagation — Live cross-tab observation / cache invalidation

**Lens:** when a user edits in one tab (tag a person, complete a task, rename a tag, write a note via MCP), does it instantly reflect in *every other* surface (other tabs, attendee chips, Today widget, the vault, the MCP) — without a manual refresh, without a re-render storm, and with caches invalidated *precisely* (not nuked)?

## Audit (through my lens)

**The good (what's already correct):**
- Core stores are `@MainActor ObservableObject` singletons injected as `EnvironmentObject` once at the root (`MeetingScribeApp.swift:34-47`, `MainWindow.swift:313-314`): `MeetingManager`, `TagStore`, `PeopleStore.shared`, `PeopleTagStore.shared`, `ActionItemStore`, `DecisionStore`. So *within* the app, a view that declares `@EnvironmentObject var people: PeopleStore` re-renders correctly when `@Published people` changes.
- `MeetingManager.refreshPastMeetings` is properly **debounced/coalesced** through a `refreshSubject` Combine pipe at 300 ms (`MeetingManager.swift:167-171, 428-444`) and additionally rate-limited to 2 s unless forced (`:432-435`). Off-main rescan, publish on main. This is the right pattern — it should be the template for everything else.
- `MeetingBodyCache` does **per-key invalidation** (`invalidate(_:)` `MeetingBodyCache.swift:121`), mtime-freshness, LRU(64), inflight coalescing, and optimistic `patchNotes/patchSummary/patchTranscript` (`:139-158`). Genuinely good cache design.
- `PeopleStore` debounces its combined-cache write at 1.5 s off-main (`PeopleStore.swift:65-74`).
- Tag *deletion* cross-propagates: `PeopleTagStore.deleteTag` → `PeopleStore.removeTagFromAll` (`PeopleTagStore.swift:127-130`, `PeopleStore.swift:1203`).

**The bugs / gaps:**

1. **MCP edits never reach the live UI.** The MCP server posts `vaultChanged` after `create_action_item`, `update_action_item`, `add_person`, person update, and `create_meeting_note` (`MeetingScribeMCP/main.swift:318, 1244, 1281, 1331, 1380, 1411`). **Nothing in the app observes `vaultChanged` or `inboxChanged`** — `grep` for `DarwinNotifier.observe(vaultChanged)` returns zero app-side hits; the only Darwin observers are recording/coreReady signals (`MeetingManager.swift:129-158`). So when Claude (via MCP) completes a task or adds a person, the open app shows **stale data until relaunch**. This breaks the "one connected workspace" promise the moment the MCP touches anything.

2. **`AttendeeChip` reads `PeopleStore.shared` statically, not as an observed dependency** (`MeetingDetailHeader.swift:682` `PeopleStore.shared.people.first {…}`, and `:735` createPerson). The struct has no `@ObservedObject`/`@EnvironmentObject var people`. So the "known-person" dot and the open-vs-create behavior **don't refresh** when a person is created/renamed elsewhere — the chip is frozen to whatever the People graph looked like at last view-build. Same anti-pattern in `MeetingSummaryTab.attendeeEmails` (`:188-190`) which reaches into `PeopleStore.shared.people` from a non-observing view.

3. **No `objectWillChange` granularity** — every store publishes whole arrays (`@Published private(set) var people: [Person]`, `items: [ActionItem]`, etc.). Tagging one person reassigns the *entire* `people` array (`PeopleStore.swift:434, 453`), invalidating every view bound to `people` — a re-render storm risk on large graphs (hundreds–thousands of records, per the load comments at `:55-60`). There is **no coalescing** on People/ActionItem/Tag publishes the way there is for meetings.

4. **Tag-rename propagation is partial.** `TagStore.onTagRenamed` renames the vault folder (`MeetingManager.swift:161-165`) but there's no equivalent "tag content changed" event to nudge cached previews or the FTS index (`PeopleStore.indexMeeting` tags column). A renamed tag can leave stale tag text in `vault_fts` and in `MeetingBodyCache` summary previews.

5. **`ActionItemStore` has no reload-from-disk path.** It decodes once in `init` (`ActionItemStore.swift:42-58`); there is no `reload()`. So even if we *did* observe `vaultChanged`, there's nothing to call to re-read `action_items.json` after the MCP rewrites it.

6. **Today widget refreshes only on `onAppear`** (`TodayView.swift:31-33` calls `refreshUpcoming()`/`refreshPastMeetings()`). Marking a task done in Tasks updates `ActionItemStore.items` (observed, fine), but a *meeting* completion or MCP write won't reflect on Today unless you leave and return to the tab.

## NET-NEW recommendations

- **SP-1 — App-side `vaultChanged` / `inboxChanged` observer that coalesces into store reloads.** *What/why:* In `MeetingScribeApp` (or `MeetingManager.init`), register `DarwinNotifier.observe(vaultChanged)` once; route into a single debounced (≈400 ms) Combine subject that fans out targeted reloads: `ActionItemStore.reloadFromDisk()`, `PeopleStore.reloadIfChanged()`, `MeetingManager.refreshPastMeetings()`. Critically, **the MCP should post a richer signal** — extend `vaultChanged` with a tiny sidecar (a `vault-change.json` "kind+id" written atomically before the post) so the app reloads *only* the affected store, not all of them. *UX:* MCP edits appear live (before→after: relaunch-to-see → instant). *Perf/stability:* one Darwin signal, debounced; sidecar keeps reload scoped (no nuke-all); off-main reads, main publish — mirrors the proven `refreshSubject` pattern. *Effort:* M. *Impact:* High. *Deps:* SP-5 (reload methods), SP-2.

- **SP-2 — A central `VaultEventBus` (typed, coalesced).** *What/why:* Replace ad-hoc string `NotificationCenter` names + missing Darwin observers with one `@MainActor` `VaultEventBus: ObservableObject` exposing a `PassthroughSubject<VaultEvent, Never>` where `VaultEvent = .person(id)|.task(id)|.meeting(id)|.tag(id)|.bulk`. Every mutator (`PeopleStore.writePerson`, `ActionItemStore` setters, `TagStore`) emits an event; the bus `.debounce(150 ms).collect()` coalesces bursts, then publishes a *de-duplicated set* of affected IDs. Views/caches subscribe to the slice they care about. *UX:* uniform, instant cross-tab sync. *Perf:* coalescing kills re-render storms; precise IDs let `MeetingBodyCache.invalidate(id)` and the FTS index update *surgically* instead of full rebuilds. *Effort:* L. *Impact:* High. *Deps:* none (foundational).

- **SP-3 — Make `AttendeeChip` and `MeetingSummaryTab` observe the People graph.** *What/why:* Add `@EnvironmentObject var people: PeopleStore` to `AttendeeChip` (`MeetingDetailHeader.swift:646`) and `MeetingSummaryTab` and resolve through it, not `PeopleStore.shared`. *UX:* tagging/creating a person in People instantly lights up the attendee chip's known-person dot + flips its tap action from create→open (before→after: stale until reopen → live). *Perf:* negligible — same data, now an observed read; SwiftUI diffs the chip only. *Effort:* S. *Impact:* Med. *Deps:* SP-7 (so it doesn't over-render).

- **SP-4 — Precise cache invalidation hooks on write.** *What/why:* `PeopleStore.writePerson` and `ActionItemStore` setters currently update in-memory + disk but don't notify derived caches. Wire each write to emit an `SP-2` event so `MeetingBodyCache.invalidate(meetingID)` fires for the specific meetings a changed person/task touches, and `vault_fts` re-upserts just that row (`PeopleStore.indexMeeting`/`upsertVaultContent`), never a full `rebuildIndex()`. *UX:* search + previews never go stale. *Perf:* surgical (one row / one cache key) vs the current "rebuild whole index" calls (`PeopleStore.swift:626, 936`). *Effort:* M. *Impact:* Med. *Deps:* SP-2.

- **SP-5 — Add idempotent `reloadFromDisk()` to `ActionItemStore` (+ `PeopleStore.reloadIfChanged`).** *What/why:* `ActionItemStore` decodes only in `init` (`:42`); add a `reload()` that re-runs `decodeArray` off-main and re-publishes, plus an mtime guard so it no-ops when files are unchanged. `PeopleStore` should gain `reloadIfChanged()` keyed on the combined-cache mtime. *UX:* prerequisite for SP-1 to actually show MCP writes. *Perf:* mtime guard avoids needless decode; off-main + main publish. *Effort:* S. *Impact:* High. *Deps:* none.

- **SP-6 — Tag-rename change event (not just folder rename).** *What/why:* When `TagStore`/`PeopleTagStore` renames a tag, emit `SP-2 .tag(id)` so (a) cached summary previews referencing the tag are invalidated and (b) affected `vault_fts` rows re-upsert their `tags` column. *UX:* renamed tag text updates everywhere live. *Perf:* scoped to rows carrying that tag. *Effort:* S. *Impact:* Low-Med. *Deps:* SP-2, SP-4.

- **SP-7 — Coalesce store publishes for large collections.** *What/why:* Bulk operations (`importPeople`, `deduplicate`, bulk-tag) currently reassign and re-sort the whole `people` array, publishing once per op — but interactive single-edits also reassign the whole array. Introduce a `withMutation { }` batching helper that suppresses intermediate `objectWillChange` and publishes once per run-loop tick. *UX:* no visible jank on a 2k-person graph when bulk-tagging. *Perf:* prevents N publishes → 1; protects the main thread. *Effort:* M. *Impact:* Med. *Deps:* none.

- **SP-8 — Today widget live refresh via the bus.** *What/why:* Subscribe `TodayView` to `SP-2` (`.meeting`, `.task`, `.bulk`) so it refreshes without relying on `onAppear` (`TodayView.swift:31`). Debounced through the same bus. *UX:* completing a task / MCP adding one updates Today instantly (before→after: re-enter tab → live). *Perf:* one coalesced subscription; reuses existing debounced `refreshPastMeetings`. *Effort:* S. *Impact:* Med. *Deps:* SP-1, SP-2.

## Top 3 picks

1. **SP-1 — App-side `vaultChanged` observer + scoped reload (Phase 1).** The single highest-value fix: today the MCP silently posts `vaultChanged` into the void, so Claude-driven edits are invisible until relaunch. Foundational/infra.
2. **SP-2 — Typed coalesced `VaultEventBus` (Phase 1).** The propagation backbone everything else hangs off; built right (debounced, ID-scoped) it *prevents* re-render storms and enables precise cache invalidation rather than causing jank.
3. **SP-3 — `AttendeeChip`/`MeetingSummaryTab` observe `PeopleStore` (Phase 2).** Cheap, high-visibility fix for a concrete stale-view bug users will hit constantly (tag a person → chip stays dumb).
